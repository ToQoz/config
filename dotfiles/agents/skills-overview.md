# Skills Overview

27 skills installed under `~/.claude/skills/`.

| 凡例 | 意味 |
|---|---|
| *(auto)* | `user-invocable: false` — スラッシュコマンドとして呼べない。Claude が自動判断して適用する |

## 開発・Git

### commit
**トリガー**: 「コミットして」「commit this」など、git commit 操作の依頼全般

Conventional Commits 形式のコミットメッセージを生成してコミットする。単なる `git commit` の代替ではなく、変更の「why」を伝えるメッセージを作ることが目的。

- `--push` フラグ: コミット後に push（保護ブランチへの push は確認を求める）
- `--pr` フラグ: push + `gh pr create` でPRを作成（`--push` を内包）
- フラグなし push の引き継ぎはしない。毎回のリクエストを独立して扱う

---

### commit-staged
**トリガー**: 「staged された変更だけをコミットして」「stage 済みの変更をコミット」など、インデックスにある変更のみをコミットしたいとき

既にステージされた変更だけを対象に Conventional Commits 形式のコミットメッセージを作成してコミットする。未ステージの変更には一切触れない。

- `--push` フラグ: コミット後に push（保護ブランチへの push は確認を求める）
- `--pr` フラグ: push + `gh pr create` でPRを作成（`--push` を内包）
- インデックスが空の場合は停止してユーザーに知らせる

---

### bash *(auto)*
**トリガー**: Bash コマンドを構築する際に自動適用

Bash コマンドを安全・正確に書くためのベストプラクティス集。Next.js の `[]`/`()` ルートや Remix の `$` パラメータなど、シェル特殊文字を含むパスを扱うときに特に有効。ユーザーが直接呼ぶものではなく、Claude が Bash を書く際に常時参照する。

---

### nodejs-package-manager
**トリガー**: Node.js プロジェクトで install/add/remove などパッケージ管理コマンドを実行する前

プロジェクトが使っているパッケージマネージャー（npm / yarn / pnpm / bun）をロックファイルから自動判定し、正しいコマンドを使う。「deps を入れて」と言われても npm を誤って使わないようにする。

---

### pr-merge *(agent)*
**トリガー**: 「PRをマージして」「CIが通ったらマージ」など

現在ブランチのPRを監視し、CI失敗を自己修正して、最終的にマージするフルワークフロー。

1. Preflight（ブランチ状態・PR確認）
2. CI 待機・失敗時の自動修正ループ
3. Codex CLI によるコードレビュー（`--skip-review` で省略可）
4. マージ実行（戦略: `--merge` / `--squash` / `--rebase` から選択）
5. ブランチ削除（`--no-delete` で抑制可）

`gh` CLI と `codex` CLI が必要。

---

### ask-codex
**トリガー**: 「セカンドオピニオンが欲しい」「高リスクな実装判断を独立検証したい」

Codex CLI (`codex exec`) に実装計画・コードレビュー・問題解決を別視点で依頼する。Codexは会話履歴を持たないため、プロンプトにすべてのコンテキストを含めた上でファイル経由で渡す（`cat file | codex exec ...`）。Claude 自身の判断と比較して、食い違いがあれば透明に報告する。

## AI / Claude 開発

## インフラ・環境設定

### nix-dotfiles *(auto)*
**トリガー**: zsh・Neovim・WezTerm・tmux・Claude Code などのdotfile設定を変更するとき（自動適用）

`~/src/github.com/ToQoz/config` を Nix/Home Manager で管理しているこの環境向けの編集ガイドライン。

- `~/.config/nvim` や `~/.claude` など symlink 先を直接編集してはいけない
- 必ず `~/src/github.com/ToQoz/config` 内のファイルを編集する
- ビルド（`darwin-rebuild switch` / `home-manager switch`）はユーザーが自分で行う

---

### mcp-builder
**トリガー**: MCP サーバーを作りたい、外部 API を LLM に繋ぎたい

LLMが外部サービスと連携するための MCP（Model Context Protocol）サーバーを高品質に構築するガイド。Python（FastMCP）と Node.js/TypeScript（MCP SDK）の両方に対応。

---

### webapp-testing
**トリガー**: ローカルWebアプリのUIテスト・動作確認・デバッグ、スクリーンショット取得

Python Playwright スクリプトでローカルWebアプリをテストするツールキット。

- `scripts/with_server.py`: サーバーのライフサイクル管理（複数サーバー同時起動対応）
- 静的HTML → DOM直接読み取り、動的Webアプリ → サーバー起動 → 偵察 → 操作 の判断フロー

---

### web-image-compression
**トリガー**: 「画像を最適化したい」「ファイルサイズを減らしたい」「WebP/AVIF に変換したい」

Web配信向け画像最適化パイプラインの構築。すべてのコマンドを `nix run nixpkgs#<package>` で実行するため、グローバルインストール不要。

- PNG 可逆圧縮: `oxipng`
- PNG 非可逆圧縮（色数削減）: `pngquant`
- JPEG 最適化: `jpegoptim`
- WebP 変換: `cwebp`
- SVG 最適化: `svgo`

## その他

### ambiguous-request-resolver *(agent)*
**トリガー**: `/ambiguous-request-resolver` スラッシュコマンドで明示的に呼ぶ

PdM・デザイナーなど非エンジニアからの曖昧なリクエストを、具体的なエンジニアリング作業に翻訳するワークフロー。

1. リクエストの翻訳（用語の意図を推定、実装詳細を推論）
2. コードベースでの根拠確認（該当UI・ルート・API・データモデルを検索）
3. 実施 vs 質問の判断（低リスクならそのまま進む、高リスクなら最小限の質問）
4. 最小限の安全な変更を実施
