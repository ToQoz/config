# Session Harness Creator

A session harness is a **disposable, task-local verification rig** generated
before implementation. It does not make the agent smarter. It makes success and
failure more observable.

> The harness is not an agent framework. It is a task-specific verification
> compiler.

This skill generates the harness package, then stops. The user runs it with
`bash .session-harness/<task-slug>/run.sh`.

## What the harness generates vs what it reuses

**Generated per task** (by LLM — the creative part):
- `prompt.md` — self-contained implementation prompt for the agent
- `verifier.sh` — mechanical checks specific to this task
- `verifier.md` — human/manual checks that can't be scripted
- `failure-modes.md` — explicit risk model
- `harness.env` — configuration (mode, iteration cap, tool permissions)

**Copied from stable runtime** (not generated — tested code):
- `run.sh` — loop mechanics, mode dispatch, logging, convergence detection

The LLM focuses on what it's good at (task analysis, check generation, prompt
crafting). Loop control flow is stable code that doesn't change per task.

## Gate: Does This Task Need A Harness?

A harness adds value when the task has **high uncertainty, high blast radius, or
verification that cannot be done by existing tooling alone**.

**Not worth it** when:
- The task follows a clear existing pattern (copy-and-adapt)
- Strong typechecking and linting already catch most failures
- The blast radius is obvious and contained
- The same agent creates and consumes the harness in one session

In these cases, tell the user and suggest a lightweight alternative (plan mode,
checklist in conversation).

**Worth it** when:
- Failure modes require runtime verification (browser, integration tests, DB)
- The task spans unclear ownership boundaries
- "Almost correct" is expensive (data loss, security, billing)
- Multiple agents or sessions are involved
- The user explicitly wants one

## Non-Negotiables

- Use only on explicit user invocation.
- Write files under `.session-harness/<task-slug>/`.
- Generate `verifier.sh` before `prompt.md` (verifier-first).
- Copy `run.sh` from `runtime/run.sh` — do not generate loop logic.
- Do not execute `run.sh`, `verifier.sh`, or invoke the agent.
- End by telling the user to run `bash .session-harness/<task-slug>/run.sh`.

## Task Package Structure

```text
.session-harness/<task-slug>/
├── run.sh              # Copied from runtime/run.sh (stable, not generated)
├── harness.env         # Task config: mode, iterations, tools, repo root
├── prompt.md           # Self-contained agent implementation prompt
├── verifier.sh         # Mechanical verification checks
├── verifier.md         # Human/manual verification checklist
├── failure-modes.md    # Risk model
├── manifest.json       # Metadata
├── state/              # Runtime output (gitignored)
│   ├── agent-N.log
│   ├── verifier-N.json
│   └── transcript.log
├── retro.md            # Generated by run.sh after execution
└── .gitignore
```

### harness.env

```bash
REPO_ROOT="/absolute/path/to/repo"
TASK_SLUG="example-task"

# Execution mode:
#   one-shot      — run agent once, verify, done
#   verify-only   — no agent, just run verifier (user implements manually)
#   bounded-loop  — verify-retry loop (default)
MODE="bounded-loop"

# Loop settings (bounded-loop only)
MAX_ITERATIONS=3

# Agent settings
# IMPORTANT: Restrict tools to the minimum needed for this task.
# Broad Bash access in an autonomous loop is a blast-radius multiplier.
AGENT_ALLOWED_TOOLS="Read,Edit,Write,Glob,Grep,Bash(make:*),Bash(git diff:*),Bash(git status:*)"
AGENT_PERMISSION_MODE="acceptEdits"
AGENT_MAX_TURNS=30
```

### Mode selection guidance

| Mode | When to use |
|---|---|
| `one-shot` | Small localized edits, strong existing tests |
| `verify-only` | User implements manually, harness only verifies |
| `bounded-loop` | Failures are cheap to detect, fixes are likely local |

Do not use `bounded-loop` when verification is weak, changes are broad, or
failures require human judgment.

### manifest.json


## Creation Phases

### 1. Triage

State:
1. Task in one sentence.
2. Why a harness helps — must name a concrete risk mitigated by executable
   verification. "It helps organize the work" is not sufficient.
3. The dominant failure that would make the task unacceptable.

Write this as the opening section of `prompt.md`.

### 2. Failure Modes

Write `failure-modes.md`: 3-8 concrete failure modes ranked by expected cost ×
probability.

Each entry needs:
- failure mode
- manifestation
- detection method
- detection type: automated or manual
- already caught by existing tooling? (tsc, lint, existing tests)

**Filter ruthlessly.** A failure mode that `tsc` catches is not a harness
failure mode — it is a compiler error. Focus on failures that **no existing tool
detects**: broken redirect flows, incorrect guard ordering, data migration edge
cases, visual regressions, cross-system integration issues.

### 3. Verifier (verifier-first)

Generate verification **before** the agent prompt. This forces clear done
criteria.

#### verifier.sh — mechanical checks

Build from four layers:

1. **Project-native checks** — existing tsc, lint, test, build commands.
   Highest signal because they encode project reality.
2. **Reusable project patterns** — check `references/patterns/` for existing
   verification snippets (auth redirects, route reachability, migration
   idempotency, etc.). Use what's relevant.
3. **Task-specific probes** — generated for this task only: grep for expected
   symbols, structural assertions, schema checks, etc. This is where the LLM
   adds unique value.
4. **Known gaps** — document what the verifier does NOT check (goes into
   verifier.md).

The script must output structured JSON and exit non-zero on failure.
See `references/templates.md` for the template.

#### verifier.md — human checks

Only for checks that genuinely cannot be automated:
- Visual/UX review
- Browser-based flow verification
- Architectural judgment

Must explicitly list **unverified risks**: "These failure modes are NOT
mechanically verified: ..."

Present a summary for user review before they run the harness:

```
Generated verifier covers:
- X (verifier.sh)
- Y (verifier.sh)

Known unverified risks:
- A (verifier.md — manual check)
- B (not covered — accept or add check)

Proceed? Edit verifier.sh first if needed.
```

### 4. Agent Prompt (prompt.md)

Write a **self-contained** implementation prompt. It must include everything the
agent needs to do the task without reading other harness files:

- Task description and goal
- Key files and patterns to follow
- Constraints and requirements
- What "done" means (matching verifier checks)

Each loop iteration gets this prompt (iteration 1) or this prompt + verifier
feedback (iteration 2+). Self-containment matters because sessions are fresh
by default.

### 5. Configuration (harness.env)

Choose mode, iteration cap, and tool permissions based on the task analysis:

- Narrow `AGENT_ALLOWED_TOOLS` to the minimum needed
- Set `MAX_ITERATIONS` based on task complexity (1-2 trivial, 3 normal, 4-5
  only with fast checks and narrow scope)
- Lower caps for destructive or external-state tasks

### 6. Copy Runtime

Copy `runtime/run.sh` from this skill's directory into the task package.
Do not modify it. The stable runtime handles:
- Mode dispatch (one-shot / verify-only / bounded-loop)
- Agent invocation via `claude` CLI
- Verifier execution and JSON parsing
- Feedback injection with task context + git diff + failed checks
- Convergence detection (repeated failure signature → early stop)
- No-diff detection (agent made no changes → early stop)
- Retro generation
- Manifest status updates

## Handoff

After generating all files, tell the user:

1. Where the harness lives: `.session-harness/<task-slug>/`
2. What the verifier checks and what it doesn't (the summary above)
3. How to run: `bash .session-harness/<task-slug>/run.sh`
4. How to adjust: edit `harness.env` for mode/iterations, `verifier.sh` for
   checks, `prompt.md` for the task description
